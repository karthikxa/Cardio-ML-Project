# Explainable Cardiovascular Disease Risk Prediction

## Abstract

Cardiovascular disease (CVD) remains one of the most important public health challenges worldwide, and earlier identification of high-risk patients can support preventive care. This project develops an explainable CVD risk prediction system for the Hack4Health Byte2Beat student health AI hackathon using public cardiovascular datasets rather than private electronic health record data. The system compares Logistic Regression, Random Forest, XGBoost, Support Vector Machine, and a feedforward deep neural network under a shared preprocessing workflow. The strongest held-out result in the local experiment was achieved by the RBF-kernel SVM with an AUC-ROC of 0.945 and F1 score of 0.914, while the neural network achieved a competitive F1 score of 0.912. For interpretability, the project uses SHAP values with the XGBoost model to rank global risk factors and explain individual patient predictions. The highest SHAP-ranked features in this run were ST slope, cholesterol, chest pain type, exercise-induced angina, and ST depression-related variables. A Streamlit application converts the trained model and SHAP explanation pipeline into an interactive clinical-style AI screening report with patient vitals, model agreement, risk-increasing factors, risk-lowering factors, and a downloadable summary. The final system demonstrates an end-to-end workflow for accurate, interpretable, and user-facing CVD risk prediction while clearly acknowledging that the model is not intended for clinical diagnosis.

## Introduction

Cardiovascular disease includes conditions such as coronary artery disease, heart failure, and stroke-related vascular disease. It is a leading contributor to mortality and long-term disability. Many CVD risk factors, including age, blood pressure, cholesterol, diabetes-related markers, smoking, exercise response, and electrocardiographic abnormalities, are routinely measured in clinical or screening settings. Machine learning can combine these signals to estimate disease risk, but healthcare applications require more than a single probability score. Users need to understand which inputs influenced a prediction and whether the model is relying on plausible clinical patterns.

This project was motivated by that gap between predictive modeling and explainable decision support. The reference repository, CVDPrediction, explored logistic regression, random forests, gradient boosting, CNNs, and LSTMs on longitudinal EHR and genetic features. That work is valuable, but its original private data cannot be used directly for a public hackathon submission. Therefore, this project adapts the model-comparison idea to public cardiovascular datasets and adds SHAP-based interpretability as the main novel contribution.

The goal is to produce a submission-ready pipeline with five practical outputs: curated public data, multiple trained models, a comparison notebook, SHAP explanations, and a Streamlit demo. This structure is designed for reproducibility and for communication with non-technical reviewers. The final system is educational, but it follows important health AI principles: separation of train and test data, prevention of preprocessing leakage, transparent metrics, model explanations, and careful language that frames the output as screening support rather than diagnosis.

## Methods

The project uses four staged datasets. The primary modeling dataset is `heart_processed.csv`, which contains patient-level heart disease features such as age, resting blood pressure, cholesterol, fasting blood sugar, maximum heart rate, ST depression, sex, chest pain type, resting ECG category, exercise-induced angina, and ST slope. The target is `HeartDisease`, binarized as 0 for no disease and 1 for disease. A larger `cardio_base.csv` dataset and its processed variant are included for later robustness experiments with the `cardio` target. A wide `ecg_timeseries.csv` file is also staged for optional temporal modeling, but the main hackathon workflow focuses on the tabular heart disease dataset because it maps cleanly to the live prediction UI.

Preprocessing is implemented in `src/preprocess.py`. The data is split into 80% training and 20% testing with stratification and `random_state=42`. Numeric features are median-imputed and standardized using `StandardScaler`. Categorical features, when present, are imputed with the most frequent value and one-hot encoded. Boolean indicators are converted to numeric values. Class imbalance is handled with SMOTE on the training set only. This is important because applying SMOTE before the train-test split would leak synthetic information into the test set and inflate performance.

The classical training script fits four classifiers: Logistic Regression, Random Forest with 200 trees, XGBoost, and an RBF-kernel Support Vector Machine. Each model is evaluated with 5-fold stratified cross-validation on the resampled training data and then evaluated on the held-out test set. Metrics include accuracy, precision, recall, F1 score, AUC-ROC, and confusion matrix. The deep learning script trains a Keras feedforward neural network with Dense layers of 128, 64, and 32 units, dropout rates of 0.3 and 0.2, Adam optimization with learning rate 0.001, binary cross-entropy loss, and early stopping on validation loss with patience 10.

Interpretability is implemented with SHAP. Although the SVM achieved the highest held-out AUC in this run, XGBoost was selected for SHAP analysis because tree-based SHAP is efficient and well suited for global and local explanations. The SHAP notebook computes values on the held-out test set, creates a beeswarm summary plot, creates a mean absolute SHAP bar chart, generates waterfall plots for individual true positive, true negative, and false positive predictions, and creates dependence plots for the top three features. The `src/interpret.py` script also compares SHAP importance with Random Forest feature importance and XGBoost gain importance. The Streamlit interface translates these explanations into a clinical-style educational report: an overall risk band, a DNN comparison check, a vitals table, risk-increasing factors, risk-lowering factors, suggested follow-up language, and a downloadable text report.

## Results

All models produced strong held-out performance on the primary dataset. The SVM with RBF kernel had the best AUC-ROC at 0.945 and the best F1 score at 0.914. The DNN achieved the same accuracy of 0.902 and a closely competitive F1 score of 0.912 with AUC-ROC of 0.936 after early stopping at 18 epochs. XGBoost achieved AUC-ROC of 0.935 and F1 score of 0.891. Random Forest reached AUC-ROC of 0.932 and F1 score of 0.898. Logistic Regression achieved AUC-ROC of 0.926 and F1 score of 0.905.

The results show that simpler and classical models remain competitive for tabular clinical data. The DNN performed well but did not clearly dominate. This is expected because the dataset is relatively small and structured, which often favors tree ensembles, support vector machines, and regularized linear models over deep learning. The strongest model by AUC was the SVM, but XGBoost remains valuable because it supports richer explanation tooling.

The SHAP analysis identified the strongest global model drivers. The highest-ranked feature was `ST_Slope_Up`, followed by `Cholesterol`, `ChestPainType_NAP`, `ExerciseAngina_Y`, and `ST_Slope_Flat`. Other important features included sex, oldpeak, atypical angina, maximum heart rate, fasting blood sugar, resting blood pressure, and age. These variables are clinically plausible because ST segment behavior, cholesterol, chest pain characteristics, and exercise-induced symptoms are all associated with cardiovascular evaluation.

The waterfall plots provide patient-level explanations. For a predicted positive case, risk-increasing features push the model output above the baseline prediction. For a predicted negative case, protective or lower-risk feature values reduce the predicted probability. The false positive example is especially useful because it shows which clinical-like signals may have caused the model to overestimate risk. This is important for responsible health AI design: errors should be inspectable, not hidden behind a probability score. In the final demo, the same idea is presented as an AI cardiovascular screening report rather than a bare prediction, improving usefulness and informativeness for Hack4Health judges.

## Discussion

The project successfully transforms a private-data reference idea into an original public-data system. The strongest part of the workflow is the combination of model comparison and interpretability. Reviewers can see not only which model performed best, but also why a tree model makes certain predictions. The Streamlit application further improves accessibility by allowing a user to enter patient vitals, receive a risk score, inspect the top contributing factors through SHAP, compare the XGBoost estimate against the neural network estimate, and download a structured screening report.

Several limitations remain. First, the primary heart dataset is small, so results can vary across random splits. A single held-out test set is useful for a hackathon demo, but stronger evidence would require external validation on multiple datasets. Second, SMOTE can help balance classes but may create synthetic examples that do not perfectly reflect real clinical populations. Third, SHAP explains model behavior rather than causal relationships. A feature that increases model-predicted risk is not necessarily a direct biological cause. Fourth, the app is educational and should not be interpreted as a diagnostic tool.

Another limitation is dataset mismatch. The local Streamlit UI includes fields such as number of major vessels and thalassemia category because those are common in Cleveland-style heart disease datasets, but the current trained local model uses the processed heart dataset fields available in `heart_processed.csv`. The app accepts those inputs for user completeness, while the deployed model currently uses the overlapping trained features. Future work can align the app with a raw Cleveland dataset pipeline or retrain on a dataset containing all planned UI fields.

Future improvements include probability calibration, subgroup fairness checks, external validation with the larger cardio dataset, and a dedicated ECG timeseries model. Calibration is especially important in risk prediction because a 70% score should correspond to an observed event rate near 70% in similar patients. The ECG dataset could support a CNN or LSTM branch, but that should be developed only after the tabular pipeline is stable.

## Conclusion

This project delivers an end-to-end explainable cardiovascular disease risk prediction system using public data. It trains and compares multiple classical and deep learning models, saves reproducible metrics, generates SHAP explanations, and provides an interactive Streamlit demo. The best held-out AUC came from the SVM, while XGBoost provided the clearest interpretability pathway. The main contribution is the practical integration of model performance, explanation, and communication: users can inspect both a global risk-factor ranking and individual patient-level drivers in a report format that feels closer to computational health screening than a classroom classifier. The system is suitable for a student health AI hackathon submission and provides a strong foundation for future validation, calibration, and temporal ECG modeling.
